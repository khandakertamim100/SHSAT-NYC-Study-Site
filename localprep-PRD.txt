# LocalPrep — Product Requirements Document (PRD)
### Version 2.0 — Updated with Practice Test Integration & Bug Fixes

---

## 1. What is LocalPrep?

LocalPrep is a free, clean, minimal web app for students preparing for
high-stakes city-specific exams. It launches SHSAT-focused but is
architected from day one to support any major city exam (TACHS, Chicago
SEHS, etc.) without rebuilding the platform.

The goal: be the best free SHSAT prep resource in NYC, then expand city by city.

## 5. Design Language

- **Vibe:** Clean and minimal — think Notion, Linear, Vercel
- **Colors:** White/off-white background, dark text, one subtle accent color
- **Typography:** Sharp, modern — nothing flashy
- **Spacing:** Generous whitespace, uncluttered
- **NO:** Gradients, neon colors, cluttered dashboards, busy animations
- **YES:** Quiet confidence, lots of breathing room, fast feel

### 6.8 Drill Mode (`/drill`)

- Immediate feedback after each answer (right/wrong + explanation)

### 6.9 Score History (`/history`)

## 7. Data Models

### Users
```
- id
- name
- email
- created_at
- google_id
```

### Exams
```
- id
- name             (e.g. "Practice Exam 1 — Form A")
- exam_number      (integer, e.g. 1 — displayed in history and session headers)
- city             (e.g. "New York City")
- total_questions  (114)
- ela_questions    (57)
- math_questions   (57)
- time_limit_seconds (10800)
- description
```

### Questions
```
- id
- exam_id          (links to Exams)
- subject          ("math" | "ela")
- topic            (specific — see Section 11 for full taxonomy)
- passage_id       (nullable — links to Passages for reading comp questions)
- passage_order    (nullable integer — question's position within a passage group)
- difficulty       (1–5)
- question_text
- choices          (array of 4–5 strings)
- correct_answer   (index 0–4)
- explanation      (text, shown after answering)
- source           ("official_doe" | "ai_generated" | "contributed")
```

### Passages
```
- id
- exam_id
- subject          ("ela")
- topic            ("reading_comp_fiction" | "reading_comp_nonfiction")
- title            (e.g. "The Development of Flower Color")
- passage_text     (full text of the reading passage)
- question_count   (number of questions linked to this passage — typically 5–6)
- line_numbers     (boolean — whether passage has numbered lines, as in official SHSAT)
```

### Test Sessions
```
- id
- user_id
- exam_id
- exam_number      (denormalized for display in history, e.g. 1)
- score            (total correct out of 114)
- ela_score        (correct out of 57)
- math_score       (correct out of 57)
- total_questions  (always 114)
- time_taken_seconds
- date
- answers          (array of {question_id, selected_answer, is_correct})
- draft_answers    (array of {question_id, selected_answer} — saves in-progress)
- time_remaining_seconds
- completed        (boolean)
- last_saved_at
```

**Mid-Test Auto-Save Behavior (unchanged from v1):**
- Draft answers and remaining time are auto-saved to Convex every 30 seconds
- If user exits mid-test, session stays with completed = false
- On return to `/test/[examId]`, check for incomplete session
- If found, show "Resume your test?" prompt with time remaining
- If resume: load draft_answers, start timer from time_remaining_seconds
- If start fresh: delete old incomplete session, create new one

### Drill Sessions
```
- id
- user_id
- exam_id
- subject
- topic
- questions_answered
- correct_count
- date
```

### Mistake Log
```
- id
- user_id
- question_id
- exam_id
- added_at
- times_seen
- times_correct    (consecutive correct answers)
- resolved         (boolean — true when times_correct hits 2)
```

## 10. Multi-Exam Architecture Notes

Every data model includes an `exam_id` and `exam_number` field. Adding a new exam is:
1. Insert a new row in the Exams table with the next exam_number
2. Add questions (and passages) tagged to that exam_id
3. It appears automatically in the test selection page and exam dropdown

No rebuilding. No refactoring. Just data.

## 13. Security Requirements (Auth Hardening)

> Acting as a senior security engineer, the following must be implemented
> before any public launch.

### 13.1 Authentication

- **Remove prototype toggle:** Delete any code that locally toggles a "signed in"
  state. All auth must flow through Convex Auth (Google OAuth).
- **Session expiry:** Sessions must expire after a configurable idle period
  (default: 24 hours of inactivity). Implement server-side session invalidation.
- **Email verification:** Enable email verification in Convex Auth settings.
  Unverified accounts should not access test data.
- **No secrets on the frontend:** API keys, session secrets, and auth tokens must
  never be exposed in client-side code or browser network requests. Store in
  Convex environment variables only.
- **Password reset tokens:** If password reset is ever added, tokens must expire
  within 15 minutes and be single-use.
- **Sign-in button centering:** The Google Sign-In button must be vertically and
  horizontally centered in the login card using Flexbox (`items-center justify-center`).

### 13.2 Rate Limiting

Apply rate limiting on all public endpoints. Use IP-based limiting for unauthenticated
routes and user-based limiting for authenticated routes.

Recommended defaults:
- Login endpoint: 10 attempts per 15 minutes per IP (return 429 with Retry-After header)
- API endpoints (unauthenticated): 60 requests per minute per IP
- API endpoints (authenticated): 300 requests per minute per user

Graceful 429 responses must include a `Retry-After` header and a human-readable message.

### 13.3 Input Validation & Sanitization

Apply schema-based validation on ALL user inputs server-side using a library such
as Zod or Convex's built-in argument validators.

Rules:
- Type checks on every argument (string, number, boolean, array)
- Length limits: question text ≤ 2000 chars, answer choices ≤ 500 chars each,
  user name ≤ 100 chars, email ≤ 254 chars
- Reject any fields not in the declared schema (strip unknown fields)
- Sanitize any text that will be rendered in HTML (escape `<`, `>`, `"`, `'`, `&`)
- Validate numeric ranges (e.g. selected_answer must be 0–4, difficulty 1–5)

### 13.4 Secure API Key Handling

- Remove any hard-coded API keys from source code immediately
- All keys must live in Vercel environment variables (server-side only)
- Convex environment variables for server functions; never passed to the browser
- No key values in `.env.local` committed to version control (add to `.gitignore`)
- Rotate any keys that were previously exposed in the codebase
